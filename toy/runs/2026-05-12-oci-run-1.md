## Findings

### Finding 1 — Toy-scale per-call latency is ~10s, not the smoke-test ~1s. `verified`

Total wall 24m34s ÷ ~140 ant-step calls (5 ants × ~9 hops avg × 3 cycles, plus loops on failures) = **~10.5s per call**. This is much closer to the smoke-test *cold* number than the warm one. The substrate primitives, evaporator, deposit hygiene all fine; the LLM call itself is the dominant cost by 2+ orders of magnitude.

### Finding 2 — Smoke-test warm 1.09s was an artifact of a trivial prompt. `verified`

The 2026-05-06 smoke test asked the queen to "Reply with exactly the word: pheromone" — ~50 char user prompt, no system prompt, 1-2 token output. The toy's ant-step prompt is ~700 chars system + ~250 chars user (path + neighbour list), with up to 20 tokens of output. On Qwen-7B-Q4 running on a 4-OCPU ARM CPU, **prompt processing dominates** — at ~50-100 tokens/sec for prompt eval on ARM, ~200 prompt tokens = 2-4s before generation even starts. The 1.09s warm number was a real measurement of a real call, but it doesn't generalise to ant-step workloads.

**Implication:** any future "ollama warm latency" claim needs to specify the prompt shape. The README's existing 1.09s number should carry a "(short prompts only)" caveat.

### Finding 3 — Queen ARM CPU is the bottleneck, not the private network. `speculative-high-confidence`

The 1.09s smoke-test number put the network hop at ~tens of ms — negligible. The bottleneck for any ant-step prompt is queen CPU compute. With `OLLAMA_NUM_PARALLEL` defaulted (likely 4), 5 concurrent ants either:

- (a) all serialize through one CPU-saturated inference (no parallelism gain), OR
- (b) actually run 4 in parallel but each takes ~4× longer because they share the same 4 cores

The wall-time evidence cannot distinguish these — both predict the same total. Worth verifying by varying `OLLAMA_NUM_PARALLEL` and ant count, but the practical conclusion is the same: **the queen as universal ant-step backend is non-viable at this hardware**.

**Implication:** TASKS.md #9 (SmolLM2-360M on ants) just promoted from "natural next" to **load-bearing for any further experiments**. The "octopus brain ↔ arm" hypothesis from CLAUDE.md — queen for hard reasoning, ants for routine ant-step — now has quantitative support, not just architectural folklore.

### Finding 4 — Qwen + concurrent ants → identical failure paths. `verified`

Cycle 2 had three failing ants with **literally identical** 13-step paths:
```
n0→n16→n15→n14→n13→n10→n11→n12→n3→n12→n3→n17→n18
```
Qwen-7B-Q4 at default temperature is deterministic. Concurrent ants firing simultaneously see effectively the same pheromone state (because each cycle's calls all queue/process before any of *their* deposits are visible). Same input → same output → same trap. This is the *opposite* failure mode from Haiku's 2026-05-05 soft anti-diversity convergence.

**Implication:** for any real experiment, ant temperature ≠ 0 is required, OR diversity needs to come from substrate-state asymmetry (each ant sees a different snapshot), OR explicit randomization in the prompt (e.g. ant-specific seed).

### Finding 5 — Qwen's failures are instruction-following soft-failures, not parse failures. `verified`

The cycle-2 failure path contains `n12→n3→n12→n3` — the LLM picked valid neighbours that were already in the path. The colony script's fallback only fires when the LLM returns an *invalid* neighbour name; revisits-of-valid-neighbours pass through. The prompt's `"avoid revisiting nodes in your path unless cornered"` is a soft instruction Qwen-7B-Q4 doesn't always follow.

**Implication:** the colony loop should harden against this — e.g. a `visited` filter applied to neighbours_with_str before showing them to the LLM (and only relax if all neighbours are visited). This is a 3-line code change.

### Finding 6 — Substrate primitives held up under real load. `verified`

37 deposit files written by 5 async-concurrent ants over 3 cycles, **zero corruption observed**, evaporator removed 8 deposits between cycle 1→2 and 41 between cycle 2→3. The ts_ns-prefixed filename scheme survived async contention.

`★` worth noting: with TAU=60s and ~10s per ant-step, deposits made early in a cycle are already at ~85% strength by the time the next ant in the same cycle reads them, and at ~5% by the time the cycle-after starts. **The temporal coupling between pheromone half-life and call latency is now load-bearing** — for a real run, TAU needs to scale with call latency, not be a hardcoded 60s.

### Finding 7 — Path diversity comparison vs Haiku is confounded. `speculative`

Qwen-7B-Q4 found *both* the 5-hop optimal path (`n0→n16→n17→n18→n19`) AND various 12-hop suboptimal paths. Haiku in the 2026-05-05 run locked onto a single 10-hop convergence path. On the surface this contradicts `feedback_llm_as_anti_diversity_prior.md`'s "LLMs over-exploit" framing — Qwen looks more exploratory.

**But:** Qwen's "diversity" includes 4 failed paths (Finding 5). It's unclear how much of the spread is genuine exploration vs incompetence at the task. Need either (a) the same comparison with Haiku on the same model temperature settings, (b) Qwen at higher temperature, or (c) a per-step instruction-following score before drawing conclusions about diversity priors.

## Implications for project direction

1. **TASKS.md #9 (SmolLM2 on ants) is now load-bearing**, not optional. The queen-as-universal-backend hypothesis is falsified at toy scale.
2. **TASKS.md #10 (substrate-distribution design)** is *less* urgent than I thought 30 minutes ago — file I/O is sub-millisecond, swamped by 10s LLM calls.
3. **`feedback_llm_as_anti_diversity_prior.md`** needs a hedge — its claim was model-specific (Haiku 4.5), not "LLMs in general". Qwen-7B-Q4 behaves differently.
4. **TAU=60s in the toy is wrong** for any non-Haiku backend. Should be parameterized as `~5× call_latency` and the README should explain.
5. **The colony script needs a visited-filter hardening** (Finding 5) — 3-line change, removes a Qwen-specific failure mode without changing behaviour for Haiku.
